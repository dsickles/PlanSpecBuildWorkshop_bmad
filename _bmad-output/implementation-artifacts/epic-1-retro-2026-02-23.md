## ⚠️ What Needs Improvement

### Pattern 1: Zod v4 API Differences (3 stories affected)

We wrote Zod v3-style code throughout Epic 1, discovered the breaking changes at review time rather than upfront:

| Story | Zod v3 | Zod v4 |
|---|---|---|
| 1.1 | `errorMap: () => ({message})` | `error: "message"` |
| 1.3 | `result.error.errors` | `result.error.issues` |
| 1.3 | `e.path: (string\|number)[]` | `issue.path: PropertyKey[]` → needs `.map(String)` |

**Fix for future epics:** Add a `### Zod v4 Notes` section to `project-context.md` or the story template Dev Notes section.

### Pattern 2: `"use server"` Semantic Misuse (Story 1.4 — Runtime Error)

`"use server"` was added to `content-utils.ts` (Story 1.2 review) and `content-parser.ts` (Story 1.3 dev notes) as a "server-only marker." This is **wrong** — `"use server"` is exclusively for **Server Actions** (async functions called from the client). It breaks non-async exports (`PINNED_PROJECT_SLUG` constant) and prevents synchronous function exports (`isError`). This caused a runtime error in the browser.

**Fix:** Use a comment (`// Server-only module`) or the `server-only` npm package. Never `"use server"` on utility modules.

**The bigger issue:** Neither ESLint nor `tsc --noEmit` caught this — it only appeared at browser runtime. This is the **most expensive bug pattern** because it slips through all static checks.

### Pattern 3: Smoke Test Debug Text Leaked to UI (Story 1.4)

The "Pipeline: 2 parsed, 0 errors" smoke test output was intentionally added to verify the pipeline, but it appeared in the visible UI and had to be manually noticed and removed by the user.

**Fix:** Smoke tests should write to `console.log` (visible only in terminal), not render to the DOM.
