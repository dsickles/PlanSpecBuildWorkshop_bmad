## 💡 Key Learnings

### Metadata Ownership & Professional Boundaries
- **Insight:** Agents attempting to "help" by auto-generated or assigning metadata (like project associations) can lead to functional inaccuracies and wasted effort if the Author has already made specific editorial choices.
- **Takeaway:** Metadata at the `/content/` level is a **High-Intent Artifact**. Agents should treat frontmatter as "Author-Only" territory unless explicitly asked to generate it. Wasted testing cycles occurred in 9.2 because the agent didn't recognize the Author's manual removal of a project association as an intentional editorial decision.

### The "Mirror" Pattern
- **Insight:** We discovered that "Pointer" files in `/content/` should be content-light and metadata-heavy.
- **Takeaway:** By moving the bulk of the prose to the root `_bmad-output/` and keeping only the filter tags in the `/content/` folder, we drastically reduced the surface area for "sync drift" between the dev environment and the portfolio.

### 🧪 Action 2: "Intent Detection" in QA
**Owner:** QA / Dev Agents
**Goal:** If a test fails because a field is "missing" but the file exists, the agent must check if the omission was a recent manual edit by the Author before attempting to "fix" it.
