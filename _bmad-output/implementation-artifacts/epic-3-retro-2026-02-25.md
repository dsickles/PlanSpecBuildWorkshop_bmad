## ✅ What Went Well

- **Performance & UX Fluidity:** The decision to use purely client-side filtering via `useMemo` and shallow routing (`{ scroll: false }`) paid off, delivering sub-100ms response times as required by NFR2.
- **Architecture Integrity:** The use of the "URL State Mapping Spec" (Action Item from Epic 2) prevented ambiguity during development. The `useFilterState` hook successfully abstracted the URL logic from the UI components.
- **Focus Mode Delivery:** The auto-expansion of blueprint groups in "Focus Mode" worked flawlessly, creating a high-value interactive experience for users.

## ⚠️ What Needs Improvement

### Pattern 1: Accessibility as an Afterthought
Initial implementions of filter pills lacked `aria-pressed` states, and icon-only buttons lacked `aria-label`. These were caught in code review but should have been part of the initial implementation.
**Lesson Learned:** Semantics and ARIA labels are core requirements, not post-implementation polish.

### Pattern 2: Global vs. Row-Level Actions (Sally's Insight)
The "Clear Filter" button lives in the "Projects" row but resets all filters. This creates spatial dissonance when only secondary filters (Domain/Tech) are active.
**Decision Deferred:** We've noted this in the [Decision Matrix](file:///c:/Users/Sickl/.gemini/antigravity/Projects/PlanSpecBuildWorkshop_bmad/_bmad-output/planning-artifacts/decision-matrix.md#4.1-Filter-Reset-UI-Positioning-(Design-Debt)). Future iterations will explore a dedicated "Active Filters" or "Action Bar" region.
